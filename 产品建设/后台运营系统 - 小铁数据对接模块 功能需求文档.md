## 一、模块定位与目标

小铁数据对接模块是后台运营系统的数据基础设施。它不直接面向运营人员提供操作界面，但后台所有上层功能模块（订单管理、异常标记、数据统计、财务对账）以及自有小程序的柜点与订单展示，全部依赖本模块从小铁同步而来的数据。本模块的稳定性与数据准确性，直接决定整个系统的可用性。

本模块的核心目标是：通过对接小铁对外数据接口，将湖滨试点范围内的机柜信息、柜门状态、订单数据，准确、及时、完整地同步至公司本地数据库，并以本地数据为基础对外提供服务，降低对小铁接口的实时依赖。

## 二、小铁接口能力总览

小铁当前对外开放四个接口，分为鉴权与数据查询两类。以下为基于真实接口定义的能力梳理。

|接口名称|路径|方法|用途|一期是否使用|
|---|---|---|---|---|
|获取 Access Token|（鉴权接口）|POST|获取调用凭证|是|
|获取机柜列表|/v1/data/cabinet/all|POST|拉取授权范围内所有机柜信息|是|
|获取机柜的柜门列表|/v1/data/cabinet/lattices|POST|拉取指定机柜的柜门（格口）详情|是|
|获取订单列表|/v1/data/cabinet/orders|POST|拉取柜门订单，支持分页与时间筛选|是|

关于接口能力的三个关键事实，直接影响本模块及整个系统的设计，必须在需求层面明确：

第一，全部接口均为只读查询接口。小铁开放的是数据同步能力，不包含下单、支付、开柜、生成取件码等写操作能力。这意味着公司系统在一期是数据的接收方与展示方，所有交易动作仍发生在小铁自有的小程序与柜机闭环内。

第二，订单接口不包含取件码字段。订单响应字段中没有任何取件码、开柜凭证类信息，因此公司后台和自有小程序无法通过现有接口获取或展示取件码。这一缺口对自有小程序的产品形态有重大影响，详见第八章。

第三，数据同步为拉取式（公司主动轮询），而非推送式。小铁不主动向公司推送数据变更，公司需要按一定频率主动调用接口拉取最新数据，这决定了本模块必须设计一套定时同步机制。

## 三、鉴权机制需求

### 3\.1 鉴权方式

所有数据接口的请求头均要求携带 Authorization 字段，格式为 Bearer token。该 token 通过"获取 Access Token"接口取得。

### 3\.2 Token 管理需求

后台系统需要实现 Access Token 的自动获取、缓存与刷新机制，避免每次数据请求都重新获取 token。具体需求如下：

系统需将获取到的 token 缓存于服务端，并在 token 有效期内复用。token 的有效期及刷新机制需以小铁 Access Token 接口的实际返回为准（该接口的返回字段需在联调阶段确认，建议在第九章接口确认清单中向小铁明确 token 有效期与刷新规则）。

当数据接口返回 401 未授权错误时，系统应自动触发 token 刷新流程并重试一次原请求，而非直接报错。这是保证同步任务稳定运行的关键容错设计。

token 属于敏感凭证，不得记录在普通业务日志中，不得在前端暴露，需按密钥管理规范存储。

## 四、机柜与柜门数据同步需求

### 4\.1 获取机柜列表

接口路径：/v1/data/cabinet/all（POST）

请求参数：

|字段|类型|必填|约束与说明|
|---|---|---|---|
|limit|integer|是|单页返回数量，取值 1 \~ 500|
|skip|integer|否|跳过条数（分页偏移），取值 0 \~ 2000|
|machine\_codes|array\[string\]|否|机柜编码数组，不传则返回授权范围内全部机柜|

响应数据（每个机柜对象 CabinetInfoToThird 的字段）：

|字段|类型|说明|本地用途|
|---|---|---|---|
|machine\_code|string|机柜编码（唯一标识）|作为本地机柜主键|
|name|string|机柜名称（当前同机柜编码）|展示用|
|address\_info|string|位置信息，格式为"网点名称\(机柜辨别地址\)"，如"广州体育馆西广场\(A\)"|柜点定位展示|
|enable\_status|string|机柜运营状态：enable（可用）/ disable（停用）|判断柜点是否可对外服务|
|online\_status|string|机柜在线状态：online（在线）/ offline（离线）|判断柜点实时可用性|
|online\_at|string|机柜上线时间|状态记录|
|offline\_at|string|机柜下线时间|异常排查|
|detect\_at|string|机柜检测时间|数据新鲜度判断|
|site\_address|object|网点地址信息（结构见下）|柜点详细地址|

site\_address（SiteAddressToThird）嵌套字段：

|字段|类型|说明|
|---|---|---|
|name|string|网点名称，如"广州体育馆"|
|province|string|省|
|city|string|市|
|region|string|区|
|region\_code|string|区号|
|address\_desc|string|详细地址|

### 4\.2 获取机柜的柜门列表

接口路径：/v1/data/cabinet/lattices（POST）

请求参数：

|字段|类型|必填|约束与说明|
|---|---|---|---|
|machine\_code|string|是|指定机柜编码|
|limit|integer|是|单页返回数量，取值 1 \~ 500|
|skip|integer|否|跳过条数，取值 0 \~ 2000|

响应数据（每个柜门对象 LockerInfoToThird 的字段）：

|字段|类型|说明|本地用途|
|---|---|---|---|
|name|string|格口编号，如"006"|格口标识|
|type|string|格口类型：small / medium / large / super\_large / other|柜型展示与价格匹配|
|available\_status|string|格口寄存可用状态：available（空闲）/ occupied（占用、使用中）/ maintenance（异常维护中）|判断格口是否可被预约/存放|
|enable\_status|string|格口运营状态：enable（启用）/ disable（停用）|判断格口是否参与运营|

### 4\.3 机柜与柜门同步逻辑需求

机柜与柜门属于相对静态的基础数据，变化频率低于订单，同步策略可与订单区分。

机柜列表建议每日定时全量同步一次，同时支持运营人员在后台手动触发同步，以应对临时新增或调整柜点的情况。

柜门列表的可用状态（available\_status）具有一定实时性，直接影响自有小程序展示给用户的格口可用情况。建议柜门数据采用较高频率的定时同步（具体频率在第五章统一说明），保证用户看到的格口状态尽可能接近真实。需要注意的是，柜门列表接口需按 machine\_code 逐个机柜查询，因此同步任务需遍历所有机柜逐一拉取。

由于柜门接口仅返回格口的可用与运营状态，不返回价格信息，格口类型（type）到价格的映射关系需要在公司本地维护一张配置表，由运营人员配置各柜型的计费规则。这是本地需要补充建设的能力，不依赖小铁接口。

## 五、订单数据同步需求（核心）

订单数据同步是本模块的核心，也是约束最多、最需要精细设计的部分。

### 5\.1 获取订单列表

接口路径：/v1/data/cabinet/orders（POST）

请求参数：

|字段|类型|必填|约束与说明|
|---|---|---|---|
|limit|integer|是|单页返回数量，取值 1 \~ 500|
|skip|integer|否|跳过条数，取值 0 \~ 2000|
|machine\_codes|array\[string\]|否|机柜编码数组，不传则返回全部|
|start\_at\_from|string|否|订单开始时间大于等于，格式 yyyy\-MM\-dd HH:mm:ss，可理解为创建时间下限|
|start\_at\_to|string|否|订单开始时间小于等于，可理解为创建时间上限|
|end\_at\_from|string|否|订单结束时间大于等于，可理解为更新时间下限|
|end\_at\_to|string|否|订单结束时间小于等于，可理解为更新时间上限|

响应数据（每个订单对象 LockerOrderInfoToThird 的字段）：

|字段|类型|说明|本地用途|
|---|---|---|---|
|order\_id|string|订单号（唯一标识）|本地订单主键，去重依据|
|status|string|订单状态：processing（进行中）/ completed（已完成）|订单状态展示与筛选|
|machine\_code|string|机柜编码|关联机柜|
|lattice\_name|string|订单使用的格口名称|定位具体格口|
|money\_paid|integer|订单支付金额，单位为分|对账、收入统计|
|start\_at|string|订单开始时间（下单/创建时间）|订单时间轴|
|end\_at|string|订单结束时间（预期结束或真正结束的更新时间）|订单时间轴|
|used\_time|integer|订单使用时长，单位分钟|计费核对|
|is\_overdue|boolean|订单是否超时|异常识别|
|overdue\_at|string|订单超时时间|异常处理|
|shop\_name|string|商户名称|多商户区分|

### 5\.2 订单接口的关键约束

订单接口存在若干硬性约束，同步机制设计必须严格遵守，否则会导致请求失败或数据遗漏。

时间范围约束：start\_at 和 end\_at 的查询范围均不超过 7 天。这意味着无法用一次请求拉取超过 7 天跨度的订单，历史数据回补或长周期同步必须按时间窗口分段循环请求。

分页约束：单页 limit 最大 500，skip 最大 2000。即单个查询条件下，通过分页最多可遍历的订单数存在上限（skip 上限 2000 加上单页 500）。当某一时间窗口内订单量较大时，需要通过收窄时间范围而非单纯翻页来获取全部数据。

金额单位约束：money\_paid 单位为分，本地存储与展示时需明确单位换算规则，避免出现金额放大或缩小 100 倍的严重错误。

### 5\.3 订单增量同步机制需求

考虑到接口约束与数据时效性要求，订单同步建议采用增量同步为主、定时全量校验为辅的策略。

增量同步方面，系统按固定频率（如每数分钟一次，具体见 5\.5）拉取最近一个时间窗口内有更新的订单。由于订单的 end\_at 可理解为更新时间，建议以 end\_at\_from 和 end\_at\_to 作为增量同步的时间游标，拉取自上次同步以来有状态变化的订单。每次同步需记录本次同步的时间游标，作为下次同步的起点。

数据落库方面，系统以 order\_id 作为唯一键进行写入。若本地已存在该 order\_id，则更新其状态、结束时间、是否超时等可变字段；若不存在，则新增。这一去重更新逻辑（upsert）是保证数据一致性的核心，必须严格实现。

兜底校验方面，建议每日定时执行一次当日全量订单的校验同步，按时间窗口分段拉取当日全部订单，与本地数据比对，修复增量同步可能遗漏的数据。

### 5\.4 状态映射需求

小铁订单状态仅有 processing 和 completed 两个枚举值，颗粒度较粗。本地系统需要建立一套状态映射规则，将小铁原始状态结合其他字段，转换为对运营和用户更有意义的本地状态。

建议的映射逻辑示例如下，具体规则需结合业务确认：当 status 为 processing 且 is\_overdue 为 true 时，本地标记为"超时使用中"；当 status 为 processing 且 is\_overdue 为 false 时，标记为"寄存中"；当 status 为 completed 时，标记为"已完成"。

这一映射的意义在于，小铁原始状态不足以支撑运营人员快速识别需要关注的异常订单，通过结合 is\_overdue 字段做本地二次加工，可以让超时这一关键异常在订单列表中直接可见。

需要明确指出的是，由于接口不提供配送、寄送相关状态，当前状态映射仅适用于寄存类订单。寄送业务的状态追踪依赖小铁新增接口，详见第八章。

### 5\.5 同步频率建议

各类数据的同步频率建议如下，最终频率需结合小铁接口的调用频次限制（需在联调阶段向小铁确认是否有限流）综合确定。

订单数据建议每 3 至 5 分钟增量同步一次，以保证订单状态在后台和自有小程序中的及时性。柜门可用状态建议每 5 至 10 分钟同步一次。机柜列表建议每日全量同步一次。当日订单全量校验建议每日凌晨低峰期执行一次。

## 六、本地数据存储设计需求

本模块需要在公司本地建立与小铁数据对应的存储结构，作为所有上层功能的数据来源。以下为存储设计的需求要点，具体表结构由开发团队细化。

机柜表需存储 machine\_code（主键）、名称、位置信息、运营状态、在线状态、网点省市区地址、上下线时间、最近同步时间等字段。

柜门表需存储所属机柜编码、格口编号、格口类型、可用状态、运营状态、最近同步时间等字段，并与机柜表建立关联。

订单表需存储 order\_id（主键）、小铁原始状态、本地映射状态、机柜编码、格口名称、支付金额（分）、开始时间、结束时间、使用时长、是否超时、超时时间、商户名称、订单来源（标识来自小铁小程序还是自有小程序）、最近同步时间等字段。

价格配置表为本地自建，需存储柜型与计费规则的映射关系，供自有小程序下单时计算价格、供对账时核对金额使用。

同步任务日志表需记录每次同步任务的执行时间、同步类型、拉取条数、成功失败状态、错误信息，供运营排查同步异常使用。

## 七、异常容错与监控需求

数据同步任务在无人值守状态下长期运行，容错与监控能力直接决定数据可信度，不能缺失。

接口调用失败时，系统需具备自动重试能力，对网络超时、5xx 服务器错误等临时性故障重试一定次数后再判定失败。对 401 未授权错误，需先刷新 token 再重试。

同步任务异常时，系统不得静默失败。当连续多次同步失败、或数据长时间未更新时，需通过告警机制（如企业微信、邮件）通知运营或技术人员，避免运营人员误以为系统正常运转而实际数据已停止更新。

数据新鲜度需可视化。后台需向运营人员展示各类数据的最近成功同步时间，使运营人员能直观判断当前看到的数据是否最新。这一点尤其重要，因为拉取式同步存在天然延迟，运营人员需要清楚知道数据的时效边界。

幂等性需保证。由于同步可能重试或重叠执行，订单写入必须保证幂等，同一 order\_id 重复同步不得产生重复记录或错误覆盖。

## 八、当前接口能力对产品的限制说明

基于真实接口能力，以下产品设计预期在一期无法仅凭小铁现有接口实现，需在产品设计中如实承认这些边界，并通过第九章的接口需求清单推动小铁补充。

取件码无法展示。订单接口不含取件码字段，自有小程序无法向用户展示取件码。这意味着用户即便通过自有小程序下单（若一期具备下单能力），其取件动作仍需回到小铁体系完成，自有小程序在一期无法形成完整的存取闭环。

寄送业务数据缺失。现有接口为寄存柜的存取订单数据，不含寄送的目的地、配送状态、物流轨迹等字段，订单状态也仅有进行中和已完成两个值。因此寄送业务的订单追踪、轨迹展示在一期无法基于现有接口实现。

下单与支付无法打通。接口为只读查询，不含下单、支付、开柜写接口。若自有小程序要实现真正的下单交易闭环，要么依赖小铁提供写接口，要么自有小程序的下单仅作为引导、实际交易仍跳转回小铁完成。这一点直接关系到自有小程序一期的产品形态定义，需与第二份 FRD（下单链路）联动确认。

用户身份信息缺失。订单接口不含下单用户的身份标识，公司无法通过该接口将订单关联到具体用户，这对自有小程序的"我的订单"功能构成限制。

## 九、需小铁新增的接口需求清单（商务推动用）

以下清单基于完整产品愿景，梳理出当前接口未覆盖、但产品闭环所必需的能力，供商务与小铁谈判时使用。按对业务的关键程度排序。

|优先级|需求接口/能力|缺失影响|业务必要性|
|---|---|---|---|
|最高|取件码获取（或下单后回传取件码）|自有小程序无法展示取件码，存取闭环断裂|自有小程序成立的前提|
|最高|下单与开柜写接口|无法在自有小程序完成真正下单交易|二期资金切换的前提|
|高|订单与用户身份的关联字段|无法实现"我的订单"，用户无法查自己的订单|用户产品基本能力|
|高|寄送业务订单与配送状态接口|寄送业务无法追踪，物流轨迹无法展示|寄送业务成立的前提|
|中|数据变更主动推送（Webhook）|只能轮询，状态更新有延迟|提升订单状态时效性|
|中|退款与售后相关接口|售后赔付、退款无法系统化处理|售后闭环完整性|
|低|接口限流规则与 token 有效期说明|同步频率设计缺乏依据|同步机制优化|

> 推进建议：上述前两项（取件码、下单开柜）是决定自有小程序能否形成交易闭环的根本，建议商务作为与小铁深化合作的首要议题。在小铁补齐这些接口之前，自有小程序的一期形态需要在第二份 FRD 中相应调整。

## 十、一期落地范围结论

综合现有接口能力，本模块在一期能够扎实落地的范围明确如下。

可完整落地的能力：机柜与柜门基础数据同步、寄存类订单数据同步、订单状态的本地映射与超时识别、基于订单金额的收入统计与对账数据基础、数据同步的容错与监控。

依赖小铁新增接口、一期暂缓的能力：取件码展示、寄送业务数据同步与追踪、自有小程序的下单开柜交易闭环、订单与用户的身份关联。

这一结论意味着，后台运营系统的数据对接、订单管理、对账等核心能力可以基于现有接口在一期顺利建成，而自有小程序的完整交易闭环受限于小铁接口缺口，其一期产品形态需要务实调整，这部分将在下一份"自有小程序下单链路 FRD"中详细定义。
