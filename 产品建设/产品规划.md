## 一、产品愿景与定位

### 1\.1 产品愿景

湖滨行李安心游的愿景，是**成为西湖景区游客行李问题的统一解决入口**，并以此为起点，构建覆盖湖滨核心商圈、酒店、三站一场的最后一公里行李服务网络，最终沉淀为可向其他景区城市复制输出的标准化能力。

愿景可以拆解为三个递进的关键词：

- **入口**：以寄存这一高频刚需为入口，统一承接游客的行李需求，聚拢流量与用户。

- **网络**：以寄送连接柜点、酒店、三站一场，把单点服务织成最后一公里网络。

- **复制**：把湖滨跑通的模式沉淀为标准化套件，向其他景区城市复制输出。

### 1\.2 产品定位

对游客而言，产品是"放下行李、轻松游湖"的统一入口——寄存与寄送在一个入口完成，全程可追踪、有人可问、存取安心。

对公司而言，产品是一张以寄存养寄送、最终可复制的行李服务网络。

**核心定位一句话：行李是切入点，网络才是目的。** 产品的长期价值不在于单笔寄存收入，而在于节点网络成型后的入口价值、数据价值与生态价值。

### 1\.3 产品的本质判断

理解这个产品，关键是理解它的双重身份：

- 它对外是一个**服务前台**（用户端小程序），解决游客的行李负担；

- 它对内是一套**数据底座**（后台运营系统），保证业务可控、数据可信、财务可对账。

更深一层，这个产品的硬件能力依托小铁的物理柜机，但业务能力完全自研。形成的核心边界是：**业务我做主，硬件靠柜控**——用户的下单意图、订单状态、支付、用户关系全由公司自研系统掌控，但任何一次真实的开关柜，都通过小铁的柜机控制接口落地。这一边界是理解整个产品架构的钥匙。

## 二、四级能力进阶模型

产品的能力建设不是一蹴而就的，而是分四级逐步进阶。每一级都建立在前一级跑通的基础上，避免越级冒进。能力进阶与商业阶段一一对应。

### 2\.1 能力进阶概览

|能级|能力定位|核心内容|对应商业阶段|
|---|---|---|---|
|L1|行李寄存工具|自研小程序 \+ 后台，提供行李寄存、取件、订单追踪、对账|验证期起步|
|L2|寄存 \+ 寄送一体服务|在寄存基础上加入寄送转运，连接车站、景区、酒店|验证期—运营期|
|L3|最后一公里行李网络|接入无人车、酒店节点、装卸中心，形成密集转运网络|运营期|
|L4|可复制的行李服务平台|标准化套件向其他城市输出，叠加 AI 行程、增值服务|复制期|

### 2\.2 各能级详解

**L1 行李寄存工具**：产品的起点。通过自研小程序与后台，提供行李寄存、取件、订单追踪、对账等基础能力，解决"存得放心、取得顺畅"，核心使命是验证游客的付费需求。这一级是工具型形态，功能聚焦、链路最短。

**L2 寄存 \+ 寄送一体服务**：在寄存基础上加入寄送转运能力，连接车站、景区、酒店等点位，产品从单点工具升级为点对点的行李服务。寄送的引入是产品价值的第一次跃升，因为它客单价更高、网络属性更强。

**L3 最后一公里行李网络**：接入无人车、酒店节点、装卸中心，形成密集的转运网络。这一级是护城河真正成型的阶段——当节点网络密集后，用户的迁移成本与对手的复制成本双双升高，产品从"服务"升级为"网络"。

**L4 可复制的行李服务平台**：将标准化套件向其他景区城市输出，叠加 AI 行程推荐、增值服务，产品从单城网络走向多城平台与生态。这一级是价值天花板的体现。

### 2\.3 进阶原则

四级进阶遵循"不越级"原则：L1 跑通（验证需求）才做 L2（加寄送），L2 跑通（验证寄送）才重投入 L3（建网络），L3 跑通（单城盈利）才做 L4（复制输出）。这一原则与 BRD 的分阶段投入决策机制一致，每一步都以前一步的真实数据为前提，控制风险。

## 三、版本规划

产品按三个版本递进发布，分别对应验证、运营、复制三个商业阶段。每个版本有明确的目标与功能边界。

### 3\.1 版本概览

|版本|名称|时间|能级跨度|核心目标|
|---|---|---|---|---|
|V1|验证版|一期·约4周上线|L1 → L2 起步|验证付费需求，建立自主入口，跑通最小闭环|
|V2|运营版|二期·自主运营|L2 → L3|入口与资金切换至自有，扩展寄送网络，盈亏平衡|
|V3|复制版|三期·标准化输出|L3 → L4|沉淀标准化套件，向其他城市复制输出|

### 3\.2 V1 验证版

**目标**：验证游客付费需求，建立公司自主入口，跑通最小商业闭环。

**功能范围**：

- 用户端：行李寄存、行李寄送（柜点存放转运）、取件码、订单追踪、客服入口、售后申请、支付；

- 后台：订单管理、异常标记、对账（过渡：数据留存与导出）、基础数据统计；

- 对接：小铁只读数据接口、小铁柜机控制接口。

**边界说明**：V1 寄送仅支持柜点存放转运，不含上门取件；支付一期走小铁收款通道；对账采用数据导出的过渡方案。V1 的核心使命不是把功能堆全，而是**跑通最小闭环、建立自主入口、用真实数据校准商业模型**。

### 3\.3 V2 运营版

**目标**：用户入口与资金切换至自有，扩展寄送网络，达成盈亏平衡。

**功能范围**：

- 资金走公司账户、自有支付商户号、数据反向推送小铁；

- 无人车配送调度、酒店节点接入、装卸中心管理；

- 会员体系、完整对账结算、数据统计大盘、寄送物流轨迹。

**边界说明**：V2 的工作重心从"产品建设"转向"支付主体切换与网络构建"，因为产品的基础能力已在 V1 建成。

### 3\.4 V3 复制版

**目标**：沉淀标准化套件，向其他城市复制输出。

**功能范围**：

- 多城市多租户、标准化部署套件；

- 广告招商平台、AI 行程推荐、增值服务、开放 API、运营数据中台。

**边界说明**：V3 是把单点经验产品化、规模化的阶段，重心在标准化与可复制性。

### 3\.5 版本关键依赖

V1 的寄存闭环依赖小铁柜机控制写接口（取件码、开柜）按时提供，这是 V1 能否如期形成闭环的**关键路径**。寄送业务是否在 V1 正式上线，取决于小铁能否提供寄送转运配套接口，否则下单界面先建、上线视能力而定。

## 五、核心用户旅程（V1）

V1 聚焦寄存与寄送两条主流程的顺畅闭环。

### 5\.1 寄存主流程

```Plaintext
选柜点 → 选柜型 → 支付 → 取件码+开柜 → 存入 → 凭码取件
```

各节点要点：

- **选柜点/选柜型**：基于本地同步数据展示可用柜点与柜型，下单锁柜时通过柜控接口实时校验，两段式防超卖；

- **支付**：一期走小铁收款通道，资金进小铁账户；

- **取件码\+开柜**：取件码由小铁柜控接口下发，用户凭码开柜；

- **存入/取件**：存入后进入寄存中状态，取件时凭码核销开柜、释放格口。

### 5\.2 寄送主流程

```Plaintext
选起点 → 选目的地 → 填收件 → 支付 → 存入起点 → 送达取件
```

寄送与寄存的差异在于：增加目的地选择与收件信息填写，且存入后进入转运而非原柜取回。一期目的地建议采用有限的固定清单，与转运能力对齐。

### 5\.3 异常路径（不可缺失）

三条异常路径必须在 V1 实现，它们是用户资金与行李安全的底线：

1. **支付成功但锁柜失败**：自动改派或原路退款，不可让用户钱卡住；

2. **取件超时补费**：取件前计算超时费用，补缴后开柜；

3. **开柜异常**：提供客服介入路径，不让用户卡在无法取件的状态。

## 六、运营策略与冷启动

产品力之外，行李服务的成败高度依赖点位密度、服务可靠与冷启动获客。规划阶段即需明确运营配套。

|维度|V1 验证期策略|V2 运营期演进|
|---|---|---|
|点位|龙翔桥等核心商圈先开 1–2 个柜点灰度|沿西湖与三站一场加密，提升网络密度|
|获客|景区现场引导 \+ 小程序码 \+ 文旅渠道|会员复购 \+ 酒店导流 \+ 品牌合作|
|服务可靠|人工兜底客服 \+ 异常工单闭环|SOP 标准化 \+ 服务质量监控|
|数据校准|跑通后校准客单价、单量、寄送占比|用数据驱动定价与网络扩张决策|

运营策略的核心判断：在 V1 验证期，点位"少而精"、服务"人工兜底"是正确选择，不必追求覆盖广度，关键是把核心动线跑通、把数据拿准。点位密度与服务可靠的规模化提升放在 V2。

## 七、衡量指标

### 7\.1 北极星指标

**周活跃寄存 / 寄送订单量**。该指标综合反映产品被真实使用的程度，兼顾寄存的频次与寄送的价值，是业务健康度最综合的体现。

### 7\.2 分阶段关键指标

|版本|核心目标|关键指标|
|---|---|---|
|V1 验证期|日均30单·付费验证|订单量、客单价、寄存/寄送结构、复购好评、自有承接占比|
|V2 运营期|日均120单·盈亏平衡|收入成本结构、节点接入数、寄送占比、网络密度|
|V3 复制期|≥1城复制签约|套件成熟度、复制城市数、启动周期、单位经济|

### 7\.3 指标的阶段侧重

北极星指标在不同阶段侧重不同：一期侧重总订单量以验证需求真实性，二期侧重自有小程序承接的订单占比以判断入口迁移效果，三期侧重可复制城市数以衡量规模化能力。

## 八、产品风险与关键依赖

|风险 / 依赖|影响|应对|
|---|---|---|
|小铁柜控接口（关键路径）|取件码/开柜依赖此接口，决定 V1 闭环能否成立|作为商务首要议题限期推进；自研主体先行不空等|
|寄送配套接口|无寄送转运接口则 V1 寄送无法追踪上线|界面先建，上线视小铁能力而定|
|点位与可靠性|点位密度与服务可靠决定口碑与留存|依托政策资源拿点位；人工兜底保可靠|
|模型假设|客单价/单量/寄送占比待验证|V1 首要任务即用真实数据校准|
|资金切换|V2 切公司账户涉及核心链路改造|V1 即采用支付主体可配置架构预留|

**关键依赖说明**：最核心的依赖是小铁柜机控制写接口。自研小程序的业务链路可完全自主，但物理开关柜依赖此接口。建议将获取柜控接口文档作为开发启动前的必备前置，优先于编码推进。其中两个问题需在拿到接口时优先确认：**取件码由谁生成**（小铁生成回传，还是公司生成后登记）、**柜机操作是同步还是异步**，这两个问题影响多个环节的实现方式。

## 九、规划要义

### 9\.1 三步走，不越级

产品按 **验证入口（V1）→ 构建网络（V2）→ 复制输出（V3）** 三步演进，对应 L1 工具到 L4 平台的能力进阶。每一步以前一步跑通为前提，用最小投入换最关键的信息，避免一次性重投入。

### 9\.2 V1 的真正使命

V1 的核心使命不是把功能堆全，而是**跑通最小闭环、建立自主入口、用真实数据校准商业模型**。最值得盯紧的，是小铁柜控接口这条关键路径，以及点位密度与服务可靠这两项决定口碑的运营变量。

### 9\.3 在文档体系中的位置

本规划承接 MRD 的市场判断与 BRD 的商业逻辑，向下统领产品需求分析与 FRD 的执行细节，是连接"为什么做"与"怎么做"的中枢。功能字段、状态机、接口交互等执行细节，详见配套的产品需求分析文档与各 FRD。
