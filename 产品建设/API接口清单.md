## 一、对接背景与协作模式

公司在湖滨街区开展行李寄存与寄送服务，自主研发用户端小程序与后台运营系统。在双方协作中，公司与小铁的接口协作分为两个层面。

数据层面，公司后台运营系统通过小铁只读数据接口，同步湖滨试点范围内的机柜、柜门、订单数据，用于运营监控、数据统计与财务对账。这部分小铁已提供接口，公司侧据此开发。

控制层面，公司自研小程序作为独立产品承载完整业务链路，但用户操作的是小铁的物理柜机，因此下单锁柜、开柜、取件码下发与核销等物理动作，需通过小铁的柜机控制写接口完成。这部分接口公司目前尚未取得文档，是本次对接的核心诉求。

简言之，公司诉求可概括为一句话：在已有只读数据接口的基础上，请小铁补充提供柜机控制写接口，使公司自研小程序能够驱动小铁柜机完成完整的存取流程。

## 二、第一类：只读数据接口（小铁已提供，确认补充项）

以下四个接口小铁已提供真实文档，公司已基于其开展后台开发。本节仅列出需向小铁进一步确认或补充的细节项，不重复接口本身。

### 2\.1 已对接接口清单

|接口|路径|用途|
|---|---|---|
|获取 Access Token|鉴权接口|获取调用凭证|
|获取机柜列表|/v1/data/cabinet/all|同步机柜基础信息|
|获取机柜的柜门列表|/v1/data/cabinet/lattices|同步柜门状态|
|获取订单列表|/v1/data/cabinet/orders|同步订单数据|

### 2\.2 需小铁确认或补充的细节项

|序号|确认项|说明与原因|
|---|---|---|
|1|Access Token 的有效期与刷新机制|文档未明确 token 有效期，公司需据此设计 token 缓存与自动刷新，避免频繁鉴权|
|2|数据接口的调用频次限制（限流规则）|公司需按限流设计订单同步频率，当前缺乏依据，担心高频同步触发限流|
|3|是否支持按区域或网点维度的数据隔离|公司仅需湖滨试点数据，需确认能否通过 machine\_codes 精确限定，避免拉取到无关数据|
|4|柜点经纬度坐标字段|现有机柜接口无经纬度，公司小程序的按距离排序、地图定位需要该字段，请确认能否提供|
|5|是否提供生产环境地址|现有文档为测试环境（test\-open\-data\.wegui\.cn），上线需生产环境地址与正式凭证|
|6|是否提供数据变更主动推送（Webhook）|现有为拉取式，状态更新有延迟，若能提供推送可显著提升订单状态时效，列为优化项|

## 三、第二类：柜机控制写接口（待小铁提供，核心诉求）

这是本次对接的重点。公司自研小程序需要驱动小铁物理柜机完成存取流程，以下能力为完整业务闭环所必需。当前公司基于行业通用智能柜控制模式做了设计，需小铁提供真实接口文档进行字段对齐。

### 3\.1 柜机控制写接口需求清单

|优先级|接口能力|用途|关键待明确点|
|---|---|---|---|
|最高|锁定格口|下单支付成功后，锁定指定柜点指定柜型的空闲格口给本订单|入参（柜机、柜型、订单标识）、返回被锁定格口标识、并发防重锁机制|
|最高|查询格口实时状态|锁柜前实时校验目标格口是否真实可用，防止超卖|是否提供实时查询、返回时延、状态枚举|
|最高|取件码下发或登记|将本订单取件凭证与格口、订单绑定|取件码由小铁柜控生成回传，还是公司生成后登记给小铁；绑定关系；有效期控制|
|最高|开启格口（存入）|用户存入行李时开启指定格口|开柜凭证形式、同步还是异步、操作结果如何回传|
|高|开启格口（取件）|用户取件时凭码开启格口|取件码核销与开柜的接口关系，是合并还是分离|
|高|取件码核销|取件开柜后使取件码失效，防重复开柜|核销时机、防重复机制|
|高|释放格口占用|用户取出行李后释放该格口，恢复可用|取出感应能力、占用释放的触发条件|
|中|存入确认（感应回传）|确认用户已将行李放入格口|是否具备存入感应，以何信号确认存入|
|中|操作结果回调或状态查询|柜机异步操作的最终结果通知|是否提供回调、回调字段、失败重试机制|

### 3\.2 两个最关键的待确认问题

在上述清单中，有两个问题会影响多个环节的实现方式，请小铁优先答复。

第一，取件码由谁生成。是小铁柜控系统生成取件码后回传给公司，还是公司系统生成取件码后登记给小铁柜机。这两种模式下，公司系统的取件码管理逻辑完全不同，需要在开发前确定。

第二，柜机操作是同步还是异步。开柜、锁柜等动作是接口调用即返回最终结果，还是先返回受理、再通过回调或轮询告知最终结果。这决定了公司系统的状态机设计与用户等待体验设计。

## 四、支付与资金协作确认项

支付环节由公司自研系统编排，但一期资金路径涉及与小铁的协作，需明确以下事项。

|序号|确认项|说明|
|---|---|---|
|1|一期资金归属与收款通道|一期用户支付资金的归属账户，以及公司自研小程序如何与约定收款方对接|
|2|计费规则对齐|公司本地价格配置需与小铁柜机实际计费规则一致，避免用户实付与展示不符，需获取小铁现行计费规则|
|3|超时费用计算规则|超时补费的计算方式需与小铁规则对齐，确保取件补费金额准确|
|4|退款通道与规则|锁柜失败退款、售后退款的资金退回路径与时效|

## 五、对账协作确认项

后台对账依赖公司本地订单数据与小铁结算数据的核对，需明确以下协议条款。

|序号|确认项|说明|
|---|---|---|
|1|对账周期|按日、按周还是按月对账|
|2|对账颗粒度|按单笔订单逐笔核对，还是按汇总金额核对|
|3|小铁结算数据的提供方式|通过接口、对账文件还是后台导出，以何种格式|
|4|差异处理责任划分|当公司订单数据与小铁结算数据不一致时的核查流程与责任归属|
|5|本地订单与小铁订单的关联标识|公司自研订单与小铁柜机订单需要可关联的共同标识，用于对账匹配，请确认以何字段关联|

## 六、需求优先级与推进建议

综合全部需求，按对项目落地的关键程度，建议推进顺序如下。

第一优先级是柜机控制写接口文档的获取，特别是锁定格口、开柜、取件码下发与核销四项核心能力，以及取件码归属、同步异步两个关键问题的答复。这是公司自研小程序能否驱动小铁柜机、形成存取闭环的根本前提，没有它下单链路无法从设计走向开发。

第二优先级是只读数据接口的补充确认项，特别是生产环境地址、token 有效期、限流规则、数据隔离能力，这些是后台系统上线运行的必要条件。

第三优先级是支付资金协作与对账协议条款的明确，这些决定财务对账模块能否定稿，可与开发并行推进，但需在上线前完成。

> 推进建议：建议优先安排一次双方技术对接会，聚焦第三章柜机控制写接口，由小铁提供柜控接口文档并答复 3\.2 的两个关键问题；同步由商务推进第四、五章的资金与对账协议条款。柜控接口文档到位后，公司可立即启动下单链路的字段对齐与开发。
