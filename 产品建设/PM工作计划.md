## 项目背景与产品范围

### 项目定位

湖滨街道行李寄存 \+ 无人车寄送场景的数字化闭环产品。

一期目标：从零搭建用户小程序（行李寄存、行李寄送、订单追踪、客服售后）及后台运营系统（对接小铁 SDK、订单管理、异常标记、对账），在 4 周内完成灰度上线，跑通最小商业闭环。

### 一期产品范围

|**纳入一期（必须做）**|**不纳入一期（二期再做）**|
|---|---|
|用户端小程序<br>✅ 行李寄存（扫码入柜 / 支付 / 取件码）<br>✅ 行李寄送（扫码入柜 / 选目的地 / 填收件 / 支付）<br>✅ 我的订单（订单列表 / 订单详情 / 物流轨迹时间线）<br>✅ 客服入口（跳转 / 在线咨询）<br>✅ 售后申请（提交入口 \+ 状态查看）<br>后台运营系统<br>✅ 后台订单看板（列表 / 筛选 / 详情）<br>✅ 异常标记与工单流转<br>✅ 对账模块（按日汇总 / CSV 导出）<br>✅ 小铁 SDK 对接（订单同步 / 状态 / 取件码）|❌ 无人车调度系统（九识对接，二期）<br>❌ 酒店节点管理<br>❌ 数据统计大盘 / BI 报表<br>❌ 会员体系 / 积分兑换<br>❌ 广告位管理<br>❌ AI 智能体 / 行程推荐<br>❌ 模式复制 SaaS 后台<br>❌ 自主收款（资金走自有账户，后期切换）|

### 两阶段支付路径说明

|**一期（当前）：资金走小铁账户**<br>用户在小铁小程序完成支付 → 资金进小铁账户 → 我方通过 SDK 读取订单数据 → 按约定周期对账结算。我方系统不承接资金流，合规风险低，落地最快。|
|---|

|**二期（切换）：资金走我方账户**<br>用户在我方自主研发的小程序完成支付 → 资金进我方账户 → 通过 API 将订单数据推送至小铁后台 → 双方系统对账。需要完成：微信支付商户号申请、小程序独立上线、API 双向推送对接。|
|---|



## 四周执行 Roadmap

### 第 1 周：筹备期 A  ——  PRD 输出 \+ SDK 解析（所有后续工作的地基）

本周目标：完成小程序 PRD 初稿、解析小铁 SDK 文档、确认支付路径，让外包团队具备动工条件。

|**周次**|**工作模块**|**负责角色**|**核心交付物**|**优先级**|**完成标志**|
|---|---|---|---|---|---|
|**W1**|小程序 PRD 输出（寄存业务）|PM|寄存用户流程图 \+ 页面清单 \+ 状态机|**最高**|外包团队确认可启动开发|
|**W1**|小程序 PRD 输出（寄送业务）|PM<br>|寄送用户流程图 \+ 目的地选择逻辑|**最高**|覆盖下单→追踪→取件全链路|
|**W1**|小铁 SDK 文档解析|PM \+ 开发|接口能力清单 \+ 缺口说明文档|**最高**|明确哪些能做哪些需绕路|

|**W1**|后台 MVP 范围说明|PM|后台功能边界文档（订单/异常/对账）|**次高**|并行启动，不让后台拖前台|

## PM 第一周行动清单（按优先级排序）

以下 6 项任务是本周不能拖延的核心工作。任何一项延误，都会导致后续节点顺延。

|**\#**|**任务**|**优先级**|**原因（不做的后果）**|
|---|---|---|---|
|**1**|**输出小程序 PRD（寄存 \+ 寄送双业务，含用户流程图 \+ 页面清单 \+ 状态机）**|**最高**|外包团队无 PRD 无法动工，延误 = 全线顺延|
|**2**|**解析小铁 SDK 文档，输出接口能力清单 \+ 缺口说明**|**最高**|决定哪些功能能做，哪些要绕路，影响开发方案|
|**3**|**书面确认支付路径：一期走小铁账户 \+ 对账结算规则（双方签字）**|**最高**|合规前提。未确认即上线，资金纠纷无依据|
|**4**|**输出后台 MVP 范围说明（订单看板 \+ 异常标记 \+ 对账三块的功能边界）**|**次高**|前台后台并行，范围不清会让后台拖累前台进度|
|**5**|**微信小程序注册 \+ 材料准备（审核周期 3\-7 天，第 2 周末必须提审）**|**次高**|错过提审时间窗口，第 4 周无法如期上线|
|**6**|**建立外包团队协作机制：日同步频次 \+ 接口约定文档 \+ Bug 分级标准**|**次高**|OPC 模式最大隐患是沟通失控，机制先于代码|



## 风险预警与应对

|**风险点**|**影响**|**应对措施**|**负责人**|
|---|---|---|---|
|**小铁 SDK 权限边界不明**|一期若不支持湖滨试点数据隔离，订单数据混淆|第 1 周内书面确认 SDK 数据权限范围；若不支持需申请独立接口授权|PM \+ 商务|
|**支付合规未落地即上线**|资金纠纷无书面依据，监管风险|对账结算规则必须双方签字后方可上线；用户协议 \+ 隐私政策法务审核|PM \+ 法务|
|**微信小程序审核被驳回**|上线时间顺延 3\-7 天|第 2 周末必须提审；预留驳回修改时间；准备好资质材料清单|PM|
|**外包 OPC 团队进度失控**|开发交付延期，影响第 3/4 周测试节点|建立每日 15 分钟同步机制；接口契约文档化；关键节点设卡点审查|PM|
|**取件码安全问题**|取件码泄露导致他人冒取行李，法律责任|取件码不得明文存储；传输全程加密；展示时限制有效期倒计时|开发 \+ PM|
|**一期需求蔓延**|外包团队工时超预算，上线延期|PRD 冻结后一期不接受新增功能；新需求进二期 Backlog|PM|



## 团队协作与决策机制

可根据实际情况再调整

### 外包团队（OPC 模式）沟通规范

- 每日 15min 站会：昨日进度 \+ 今日计划 \+ Blocker（异步文字同步）

- 接口约定文档化：前后端接口不依赖口头约定，全部写进接口文档

- Bug 分级标准：P0（阻断上线）/ P1（有绕路方案）/ P2（优化项）

- 版本管理：Git \+ 分支规范，每个功能模块独立分支，PM review 后合并



### PM 本人工作节奏

- 产出优先：PRD / 文档 / 决策纪要 优先于会议

- 每日结束前：记录当日决策 \+ 明日重点，同步项目群

- 每周五：输出本周进度小结（对内汇报 \+ 对外公示）

- 上线后 72h：全天候跟进，每 4h 一次数据快报



### 与小铁团队对接节奏

- 第 1 周：SDK 权限确认会（30min，书面纪要）

- 第 2 周末：联调启动（技术对接，有记录）

- 第 3 周：压测验收（输出联调报告）

- 每周：财务对账核对（对账记录留存备查）
