## 一、项目背景

杭州湖滨街道计划在三站一场（火车站、汽车站、机场、体育场馆周边）及湖滨核心商圈试点行李寄存与寄送服务，目标是用行李这一刚需场景，构建游客的"最后一公里"行李解决方案，同时为后续无人车配送、酒店节点接入等更大范围的商业闭环打基础。

公司在该项目中承担产品方与运营方的角色。项目采取分阶段推进策略，但需要特别厘清的是，第一阶段内部实际包含两条并行且彼此独立的工作线，二者不存在依赖关系，时间上同步推进、系统上各自独立：

第一条工作线是业务验证。业务验证完全依托小铁（深圳市沐腾科技有限公司）现有的"小铁寄存柜"小程序进行，用户在小铁小程序内完成下单与支付，资金进入小铁账户。公司侧通过对接小铁 SDK，同步获取本项目在湖滨街区试点范围内的订单数据，建设一套后台数据系统，实现订单可查看、可对账、可追踪。这条线的目的是借助小铁成熟的现有能力，让业务尽快上线运转、快速沉淀真实经营数据。

第二条工作线是自有用户端小程序建设。公司同步独立开发自有用户端小程序，该小程序不依赖、也不嵌入小铁小程序，是公司完全自主的产品。根据业务方确认，自有小程序在第一阶段内即完成开发并正式上线对外，与小铁小程序并行向用户提供服务。这条线是为了让公司尽早建立自己的用户入口与产品能力，为第二阶段全面承接用户与资金做准备。

第二阶段的目标是将寄存与寄送两大业务的用户入口完全切换至公司自有小程序，用户支付资金流转入公司账户，公司系统反向通过 API 将订单数据推送至小铁后台，完成双方的对账协同。

需要明确的是，"资金流转入公司账户"是第二阶段才发生的切换动作。因此在第一阶段，自有小程序虽然上线对外，但其支付资金的归属仍需与业务方最终确认（详见第八章待确认事项），本文档基于"第一阶段资金统一进入小铁账户"这一与项目原始规划最一致的假设展开后续分析。

## 二、产品定位

行李安心游不是一个独立的工具型产品，而是公司在湖滨街区行李场景下的服务前台与数据底座。它对外呈现为一个解决游客行李负担的小程序，对内则是一套保证业务可控、数据可信、财务可对账的运营系统。

由于自有小程序在第一阶段即上线对外，第一阶段的产品形态已经是一个面向真实用户的服务产品，而非单纯的内部管理工具。这一点与上一版本的判断不同，也直接提升了第一阶段产品设计的完整度要求。

产品的核心价值主张可以归纳为三句话：

对用户而言，行李寄存和寄送在一个入口内完成，全程可见、可追踪、有人可问。

对运营团队而言，所有订单状态、异常情况、资金流水都能在一个后台里看清楚，不需要依赖第三方系统的原始界面去拼凑信息。

对公司而言，无论当前业务依托哪一方的支付通道，订单数据和用户关系始终沉淀在自己的系统里，这是从代运营走向自主运营的必经路径。

## 三、用户角色与场景分析

### 3\.1 用户角色画像

本项目涉及四类核心角色，他们的需求出发点完全不同，必须分开分析，不能用一套需求覆盖所有人。

游客用户是终端使用者，典型场景是携带行李在西湖周边游玩，希望摆脱体力负担，对价格敏感度中等，对服务可靠性的敏感度很高，一旦行李出现问题会直接影响对整个景区的体验评价，因此他们最核心的诉求是安全感，其次才是便捷。

柜点工作人员或无人柜本身是服务的执行节点，在第一阶段完全由小铁的硬件和系统承担，公司不直接参与柜机层面的运维，这部分对公司而言是黑盒，只需要通过 SDK 获取状态结果。

运营人员是公司内部负责日常订单监控、异常处理、客诉响应的角色，他们的核心诉求是信息集中、操作简单、责任可追溯，不希望在多个系统之间来回切换查证据。

财务对账人员关注的是订单数据与资金流水能否对得上，第一阶段尤其需要解决的问题是公司系统里的订单记录与小铁实际收款记录之间的口径是否一致，这是最容易出现纠纷的环节。

### 3\.2 核心使用场景

游客的核心场景可以拆分为两条主线。

第一条主线是寄存场景：游客到达湖滨某个柜点，希望临时存放行李，扫码或在小程序内下单，选择柜型，完成支付，将行李存入柜内，获得取件码，游玩结束后凭码或人脸识别取出行李。这条主线的关键体验节点在于"存得放心、取得顺畅"，任何环节卡顿都会被放大成负面体验。

第二条主线是寄送场景：游客不想随身携带行李，希望将行李从当前位置直接送达另一个地点，例如从景区送到酒店，或从酒店送到车站。根据业务方确认，第一阶段的寄送业务仅支持柜点存放转运模式，即用户将行李存入指定柜点后由系统完成站点间转运，暂不包含上门取件能力。这一边界明确降低了第一阶段寄送业务的流程复杂度。寄送场景中用户最关心的是"我的行李现在在哪里、什么时候能到"，因此物流轨迹的实时性和准确性是这条主线的核心难点。

两条主线在异常情况下都会衍生出售后场景，包括超时未取、行李损坏、行李遗失、配送延误等，这部分往往是产品口碑的分水岭，需要在需求阶段给予足够重视，而不是当成边缘功能简单处理。

## 四、产品需求详细分析

本章节按照自有小程序端和后台运营系统两个子系统分别展开，每个子系统内部按照核心模块逐一分析需求逻辑与设计要点，而非简单列出功能清单。

需要说明的是，第一阶段的业务验证发生在小铁小程序上，公司不对小铁小程序本身做任何开发，因此本章节针对自有小程序端的需求分析，对象是公司将在第一阶段独立开发并上线的自有产品。

### 4\.1 自有用户端小程序需求分析

#### 4\.1\.1 行李寄存模块

行李寄存的核心设计难点在于柜机信息的实时性。如果小程序展示的柜点状态与实际柜机可用格口不一致，用户到现场后无法顺利存放，会直接损害信任。由于自有小程序的柜机状态数据来源于小铁 SDK，因此该模块的需求不仅是做一个下单界面，还要在产品逻辑上明确柜点状态的更新频率、缓存策略与异常兜底方案，例如当某个柜点格口被占满时，前端应当如何提示用户改选其他临近柜点。

下单流程需要覆盖选择柜点、选择柜型规格、确认价格、完成支付、获得取件码这一完整链路。其中价格展示需要考虑按小时计费还是一次性计费的规则差异，这部分需要与小铁的计费逻辑对齐，避免出现公司展示价格与实际扣款不一致的情况。

取件码是寄存场景里最核心也最敏感的信息，需求上必须明确取件码的生成来源（来自小铁 SDK 而非公司自行生成）、展示形式（数字加二维码双重形式，兼顾不同年龄段用户的使用习惯）、有效期管理（超时未取的提醒机制与后续处理流程）。

#### 4\.1\.2 行李寄送模块

第一阶段寄送业务仅支持柜点存放转运模式，这一确认显著简化了本模块的需求设计。用户的寄送流程为：选择目的地、填写收件信息、将行李存入指定起始柜点、完成支付、等待系统转运至目的地。由于不涉及上门取件，本阶段无需设计取件人员调度、上门时间预约等复杂流程，相关人力调度类需求可整体推迟至后续阶段评估。

寄送场景的下单确认环节，需要向用户清晰展示预计送达时间窗口，而非模糊的"尽快送达"。这个时间窗口的可信度直接取决于后台数据同步的及时性，因此该需求实际上对后台系统的数据同步频率提出了硬性要求，前后端需求在这里是耦合的，不能割裂设计。

#### 4\.1\.3 我的订单模块

订单追踪是整个产品里用户最频繁查看、也最容易产生焦虑情绪的模块，需求设计的核心原则是信息透明优于信息丰富。与其展示复杂的物流节点图，更重要的是用清晰的文字状态告诉用户行李现在处于哪个阶段，距离上一次状态更新过去了多久。

需要特别关注的一个需求点是状态颗粒度问题。如果后台从小铁 SDK 拿到的状态只有寥寥几个大类（比如已入柜、配送中、已送达），而前端却试图展示更细的物流轨迹，会出现数据与展示不匹配的问题，造成虚假的精细化体验。这部分需求必须在与小铁 SDK 对接确认接口能力后，倒推前端展示颗粒度，而不是先设计理想化的前端再去找数据支撑。

#### 4\.1\.4 客服咨询模块

客服模块在第一阶段的需求边界应当务实地定义为"咨询入口"而非"智能客服系统"。考虑到一期资源有限，建议优先打通人工客服渠道（微信客服或在线客服第三方工具），明确响应时间承诺，而不是投入资源做自动化应答机器人。客服入口需要在订单详情页和异常状态页面有清晰可见的入口，而不能让用户在遇到问题时找不到求助路径，这是用户体验里最容易被忽视但后果最严重的细节。

#### 4\.1\.5 售后服务模块

售后模块的需求设计需要先明确处理范围的边界，常见的售后场景包括超时未取产生的额外费用争议、寄送过程中的延误、行李损坏或遗失的赔付申请。第一阶段的售后流程建议设计为用户提交申请、人工审核处理、结果通知三个环节，不追求自动化判定，因为行李类纠纷往往涉及主观判断和现场证据，自动化规则容易引发用户不满。

售后模块需要与后台运营系统的异常工单功能形成闭环，用户在前端提交的每一个售后申请，都应当在后台生成对应的工单记录，并可追踪处理进度，这是保证服务质量可控的关键设计点。

#### 4\.1\.6 支付模块

由于自有小程序在第一阶段即上线对外，支付模块成为第一阶段必须落地的能力，这是本版本相较上一版本新增的重要需求模块。

支付模块需求的核心不确定性在于第一阶段资金的归属。本文档基于"第一阶段资金统一进入小铁账户"的假设展开，即自有小程序在第一阶段通过对接小铁的收款通道完成支付，资金进入小铁账户，第二阶段再切换为公司自有商户号收款。基于这一假设，第一阶段支付模块的需求重点是与小铁收款通道的对接，而非自建支付清结算能力。

无论最终确认采用哪种资金归属方案，支付模块在架构设计上都应当采用支付主体可配置的思路，为第一阶段到第二阶段的支付主体切换预留扩展空间，避免二期改造时对核心交易链路进行大幅重构。

### 4\.2 后台运营系统需求分析

#### 4\.2\.1 订单管理模块

订单管理模块是整个后台系统的信息中枢，需求设计的核心是回答一个问题：运营人员能否在不打开小铁原始后台的情况下，完成日常的订单查看与问题定位。这要求订单列表展示的字段必须覆盖运营人员判断订单状态所需的全部关键信息，包括订单编号、用户标识、柜点位置、当前状态、创建时间、最近更新时间、金额、订单来源（区分订单来自小铁小程序还是自有小程序）。

订单来源字段在本项目中具有特殊意义。由于第一阶段存在小铁小程序和自有小程序两个入口并行运行，后台必须能够清晰区分每一笔订单来自哪个入口，这既是运营管理的需要，也是后续分析自有小程序导流效果的数据基础。

筛选与搜索能力的需求优先级应当高于美观的数据可视化，因为运营人员的日常工作场景是快速定位某一类问题订单（例如所有超时未取的订单），而不是浏览整体趋势图表。第一阶段建议把统计图表类需求降级，优先保证筛选查询的效率。

#### 4\.2\.2 异常标记与处理模块

异常标记模块的需求设计需要先建立一套异常分类标准，常见分类包括配送延误、行李损坏、行李遗失、取件码错误、支付异常等。每一类异常应当对应不同的处理流程和责任归属，这部分需要提前与业务方和法务确认责任划分原则，避免运营人员在处理现场纠纷时无据可依。

工单流转状态的设计需要覆盖待处理、处理中、已处理、已关闭这几个基本状态，并允许运营人员添加处理备注，形成完整的处理记录留痕，这不仅是运营管理需求，也是潜在纠纷场景下的重要证据链。

#### 4\.2\.3 数据统计模块

数据统计模块在第一阶段的需求应当保持精简，核心目的是支撑运营决策和对外汇报，而非追求完整的商业智能分析能力。建议聚焦在日订单量、日收入、柜点使用率、异常率这几个最基础但最能反映业务健康度的指标上。

考虑到第一阶段两个入口并行的特殊情况，数据统计模块建议增加按订单来源拆分的维度，使管理层能够直观看到自有小程序上线后的实际承接量与小铁小程序的对比，为第二阶段全面切换入口的决策提供数据支撑。复杂的多维交叉分析、用户行为分析建议放入二期规划。

#### 4\.2\.4 财务结算与对账模块

财务结算与对账模块是本项目第一阶段最容易被低估难度、却最容易出问题的模块。其需求复杂度的根源在于，公司系统记录的订单数据来自小铁 SDK 的同步结果，而实际资金流水发生在小铁自己的账户体系内，两套数据之间存在天然的核对需求。

需要特别指出的是，截至本文档编制时，公司与小铁之间的对账周期、对账颗粒度、差异处理责任划分等具体协议条款尚未最终确定。这意味着本模块的详细需求目前只能基于假设展开，存在后续返工的风险。对账协议条款的确定，是本模块需求能够真正定稿的前置条件，建议作为重点风险事项推动业务方与商务尽快敲定。

在协议条款明确之前，本模块的需求设计应当保持灵活性，至少需要支持订单数据的完整留存与导出能力，使财务人员在系统对账能力尚未完善时，仍能通过导出数据与小铁结算单进行人工核对，作为过渡方案。

#### 4\.2\.5 小铁数据对接模块

这个模块本质上不是面向运营人员的功能模块，而是整个后台系统的数据基础设施。由于它直接决定了上述所有模块数据的真实性和及时性，必须作为独立模块单独分析需求。

根据业务方确认，小铁 SDK 支持按区域进行数据隔离与权限控制，这一确认消除了上一版本中最主要的需求风险，意味着公司系统能够准确获取湖滨试点范围内的订单数据，而不会混入小铁全国其他城市的数据。这为订单管理、数据统计、对账等所有上层模块的需求设计奠定了可靠的数据基础。

本模块剩余的核心需求点包括：从小铁 SDK 同步订单数据的频率与方式（实时推送还是定时轮询，这直接影响订单追踪模块的用户体验）、数据字段的映射关系（小铁系统的字段命名和业务含义与公司系统如何对应，避免理解偏差导致的展示错误）、异常容错机制（当小铁系统出现接口异常或数据延迟时，公司系统应当如何提示运营人员，而不是静默失败让运营人员误以为系统正常）。

## 五、阶段性需求边界划分

明确划分两个阶段的需求边界，是避免本项目出现需求蔓延和重复开发的关键。本章节根据业务方确认的信息重新梳理。

### 5\.1 第一阶段需求边界

第一阶段包含两条并行独立的工作线，二者互不依赖，同步推进。

业务验证线方面，业务完全运行在小铁现有的小铁寄存柜小程序上，用户在小铁小程序内下单与支付，资金进入小铁账户。公司侧的工作是建设后台数据系统，通过 SDK 同步湖滨试点范围内的订单数据，实现订单查看、状态追踪、对账核验。公司不对小铁小程序做任何开发。

自有产品线方面，公司独立开发自有用户端小程序，覆盖行李寄存、行李寄送（仅柜点存放转运）、订单追踪、客服咨询、售后服务、支付六大模块，并在第一阶段内完成开发并正式上线，与小铁小程序并行对用户开放。自有小程序的柜机状态、订单状态等数据同样来源于小铁 SDK，但产品入口与体验完全由公司自主掌控。

因此，第一阶段的完整交付物为：一套对接小铁 SDK 的后台运营系统（订单管理、异常处理、数据统计、财务对账、数据对接），以及一个独立开发并上线的自有用户端小程序。两个入口并行运行带来的用户引流、体验区分等运营层面的问题，需要在运营方案中专门设计，本文档已在需求层面通过订单来源区分等设计点予以支撑。

### 5\.2 第二阶段需求边界

第二阶段，公司自有小程序正式成为寄存和寄送两大业务的主要用户入口，用户支付资金切换为进入公司自有账户。此时公司系统的角色发生反转，从数据接收方变为数据输出方，需要将订单数据通过 API 反向推送给小铁后台，以保证小铁仍能掌握其柜机和运力资源的实际使用情况，支撑双方的协议结算。

第二阶段新增的核心需求包括公司自有微信支付商户号的接入与资金管理、数据反向推送接口的开发与稳定性保障，以及小铁小程序入口的逐步收敛策略。由于自有小程序的产品能力已在第一阶段建成并上线，第二阶段的工作重心从产品建设转向支付主体切换与数据流向重构，建议作为独立的二期项目单独立项规划。

## 六、关键需求依赖与风险点

经过本轮边界确认，本项目的需求风险较上一版本已显著收敛，但仍存在若干需要重点关注的依赖与风险。

第一阶段两个入口并行运行是本版本需要新增关注的运营与数据风险点。小铁小程序与自有小程序同时面向用户开放，可能带来用户体验割裂、订单分流口径混乱等问题。需求层面已通过后台订单来源区分予以支撑，但运营层面如何引导用户、如何避免两个入口的服务标准不一致，需要在运营方案中专门设计。

财务对账协议条款尚未确定，是当前最主要的需求阻塞点。由于公司与小铁之间的对账周期、对账颗粒度、差异处理责任划分尚未敲定，财务对账模块的详细需求无法最终定稿，存在返工风险。建议在系统设计阶段优先保证订单数据的完整留存与导出能力作为过渡兜底，同时推动商务尽快完成协议条款的确认。

取件码等敏感信息的数据安全要求仍需高度重视。无论是同步展示小铁生成的取件码，还是涉及支付环节的资金信息，都需要明确传输加密、存储规范、展示控制的安全需求。这部分需求容易被当作普通字段处理，但实际上涉及用户财产安全，需求优先级应当被提升。

支付主体切换对产品架构的影响需要在第一阶段架构设计时就予以考虑。第一阶段到第二阶段存在支付资金归属的切换，如果产品架构没有为这一切换预留扩展空间，会导致二期改造时需要对核心交易链路进行大幅重构。建议第一阶段即采用支付主体可配置的架构思路。

需要说明的是，上一版本中列为最高风险的"小铁 SDK 数据隔离能力不明"问题，已通过业务方确认小铁 SDK 支持按区域数据隔离而消除，不再构成需求风险。

## 七、需求优先级建议

综合以上分析，本项目第一阶段的需求优先级建议如下排序，排序依据是该需求对业务闭环成立的必要程度，而非开发难度。由于两条工作线并行，优先级在两条线内部分别说明。

后台数据系统方面，最高优先级是小铁 SDK 数据对接能力的开发，因为这是整个后台系统乃至自有小程序数据展示的共同基础。次高优先级是订单管理模块与异常标记模块，因为这两个模块直接关系到运营团队能否在业务上线后实际接管日常运营工作。财务对账模块优先级略低，但鉴于其协议条款尚未确定，建议第一阶段优先实现数据留存与导出的过渡能力，完整对账能力待协议明确后补齐。数据统计模块优先级最低，第一阶段保持最小化实现即可。

自有小程序方面，由于需要在第一阶段内完成开发并上线，整体优先级较高。其中寄存与寄送的下单主流程、支付模块、订单追踪模块属于上线必备的核心能力，优先级最高。客服与售后模块属于保证服务闭环的必要能力，优先级次之，但在上线前必须具备基本可用形态，不能缺失。

两条线之间，建议在资源分配上优先保证后台 SDK 对接能力与自有小程序核心下单链路的并行推进，因为这两者分别是两条线能否成立的根本，其余模块可在此基础上逐步补齐。

## 八、待确认事项

经过本轮确认，上一版本的多数待确认事项已经明确。当前仅保留以下两项需要业务方进一步确认，其余事项已在前文相应章节落实。

第一阶段自有小程序上线后的支付资金归属，是当前最重要的待确认事项。本文档基于"第一阶段资金统一进入小铁账户、第二阶段再切换至公司账户"的假设展开。若业务方实际期望自有小程序一上线即采用公司自有账户收款，则第一阶段就需要完成公司微信支付商户号的申请与自建支付清结算能力的开发，第一阶段的工作量和合规要求将显著增加，需相应调整需求范围与排期。此项必须在支付模块详细设计启动前明确。

公司与小铁之间的对账周期、对账颗粒度、差异处理责任划分的具体协议条款，是财务对账模块需求定稿的前置条件。在协议条款明确之前，相关需求只能基于假设展开，建议推动商务尽快完成协议谈判与条款确认。
