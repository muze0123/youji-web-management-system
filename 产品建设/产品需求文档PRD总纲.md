## 一、产品概述

### 1.1 产品背景

湖滨行李安心游是面向西湖景区游客的行李寄存与寄送一体化服务。一期目标是从零搭建公司自研的用户端小程序与后台运营系统，依托小铁（深圳市沐腾科技）的物理柜机网络，跑通最小商业闭环，用真实数据验证游客付费需求。

核心边界：**业务由公司自研系统掌控，物理柜机操作通过小铁柜机控制接口落地**。

### 1.2 产品目标（V1）

| 目标 | 衡量标准 |
|------|----------|
| 跑通行李寄存与行李寄送最小闭环 | 两条主流程端到端可用，0 个 P0 缺陷 |
| 建立公司自主入口 | 自研小程序上线对外，沉淀用户与订单数据 |
| 验证付费需求 | 灰度达成日均 30 单，获取真实客单价与单量 |
| 后台可运营可对账 | 运营团队能查看订单、标记异常、完成对账 |

### 1.3 范围

**V1 纳入**：用户端小程序（登录、行李寄存、行李寄送、订单追踪、取件码、客服、售后、支付）；后台（订单管理、异常工单、对账过渡、基础统计、运营配置）；对接层（小铁数据同步、柜机控制、支付与通知）。

**V1 不纳入**：上门取件、配送网络规模化、无人车调度、资金切换公司账户、自建清结算、数据反向推送、会员体系、数据大盘、酒店节点、多城市多租户。

### 1.4 概念层级说明（开发须知）

为避免歧义，明确以下资源层级：

| 概念 | 定义 | 标识 |
|------|------|------|
| 柜点 / 网点 | 一个地理位置，可含一个或多个机柜 | 网点名称 |
| 机柜 | 一台物理寄存柜 | 机柜编码 machine_code |
| 格口 | 机柜内的单个储物格，有柜型与可用状态 | 格口编号 name（如 "006"） |
| 柜型 | 格口的尺寸类别 | small/medium/large/super_large/other |

**选择逻辑**：用户在"选柜机"阶段选定一台机柜；在"选格口"阶段按柜型选择所需规格（系统在锁柜时分配一个该柜型的具体空闲格口，用户无需手动指定格口编号）。

### 1.5 术语

| 术语 | 定义 |
|------|------|
| 行李寄存 | 游客临时存放行李，按存放时长计费，超时补费 |
| 行李寄送 | 游客将行李点对点转运至另一地点，一期仅柜点存放转运 |
| 存放时长 | 下单时预选的寄存时长，决定预付费用 |
| 取件码 | 取件凭证，由小铁柜控接口下发，凭码开柜 |
| 柜机控制接口 | 小铁写接口，锁柜/开柜/取件码下发核销（待小铁提供） |
| 只读数据接口 | 小铁已提供的查询接口，同步机柜/柜门/订单 |

## 四、全局设计规则

### 4.1 行李寄存订单状态机

| 状态 | 含义 | 流转 |
|------|------|------|
| 待支付 | 已创建未支付 | 支付成功→已支付待锁柜；超时/取消→已取消 |
| 已支付待锁柜 | 支付成功等待锁柜 | 锁柜成功→寄存中；锁柜失败→退款中 |
| 寄存中 | 行李已存入正常寄存 | 发起取件→待取件结算；超出预选时长→超时寄存中 |
| 超时寄存中 | 超过预选时长未取 | 发起取件→待取件结算（含补费） |
| 待取件结算 | 取件前计算补费 | 补费成功/无需补费→已完成 |
| 已完成 | 取件完成格口释放 | 终态 |
| 已取消 | 未支付或主动取消 | 终态 |
| 退款中 | 锁柜失败或售后退款 | 退款成功→已退款 |
| 已退款 | 退款完成 | 终态 |
| 异常 | 需人工处理 | 处理后回对应态 |

### 4.2 行李寄送订单状态机

| 状态 | 含义 |
|------|------|
| 待支付 | 已创建未支付 |
| 已支付待锁柜 | 支付成功等待锁定起点格口 |
| 待转运 | 行李已存入起点等待转运 |
| 转运中 | 转运过程中（状态来源依赖转运接口） |
| 已送达待取 | 已到目的地等待取件 |
| 已完成 | 取件完成 |
| 已取消/退款中/已退款/异常 | 同行李寄存规则 |

### 4.3 取件码规则

由小铁柜控接口下发或登记（生成方待接口确认，做两套兼容）；与订单、格口绑定，有有效期；加密传输、不明文存储；取件核销后失效防重复开柜；数字+二维码双重展示。

### 4.4 计费规则（含存放时长）

- 价格由公司本地价格配置表计算，不依赖小铁；按柜点、柜型维度配置（时长档单价、起步价、封顶价、超时费率）；
- **行李寄存计费模型**：用户下单时选择**存放时长档**（如 4h / 8h / 24h / 自定义），据此预付费用；实际取件若超出预选时长，按超时费率在取件前计算补费；
- 金额内部以"分"存储，展示换算为元；
- 本地配置需与小铁柜机实际计费对齐，避免实付与展示不一致；
- 行李寄送计费按转运服务费，按目的地/距离档配置。

### 4.5 权限角色（后台）

| 角色 | 权限 |
|------|------|
| 管理员 | 全部功能，含运营配置、权限管理 |
| 运营员 | 订单查看、异常处理、对账查看，不含配置与权限 |

### 4.6 异常处理总则（三条 P0 底线）

1. **支付成功但锁柜失败**：自动尝试同柜型其他空闲格口；无可用则引导改选其他柜型/邻近柜机，或原路退款，不让用户钱卡住；
2. **取件超时补费**：取件前按实际时长与超时费率计算补费，补缴成功后开柜；
3. **开柜异常**：开柜失败、码失效等提供客服介入路径，不让用户卡在无法取件状态。

### 4.7 数据安全规则

取件码、支付信息加密传输，取件码不明文存储；手机号脱敏（138\*\*\*\*8888）；不存支付卡信息；个人信息最小必要与合规授权。

### 4.8 UI 设计基调（原型指引）

- **平台**：用户端为微信小程序（竖屏，约 375–414pt 宽）；后台为 Web 管理界面。
- **调性**：干净、可信、轻松。核心情绪是"安心"，视觉避免廉价感，突出安全与清晰。
- **色彩**：主色建议与西湖气质契合的青绿色系，辅中性灰与暖白，关键操作用高对比强调色；状态用色区分（进行中/完成/异常/退款）。
- **组件**：卡片式信息组织、主次按钮清晰、状态用 Badge、金额突出、取件码大号易读。
- **文案**：简洁直白，减少焦虑；异常与等待场景给明确指引。

## 六、核心流程总览

### 6.1 行李寄存主流程

```
选柜机 → 选格口(柜型) → 选存放时长 → 订单确认 → 支付
   → 锁柜(防超卖) →[锁柜失败兜底]→ 下发取件码
   → 扫码开柜存入 → 寄存中 →[超时→超时寄存中]
   → 发起取件 →[超时补费]→ 核销开柜 → 释放格口 → 已完成
```

### 6.2 行李寄送主流程

```
选起点柜机 → 选起点格口 → 选目的地 → 填联系信息 → 填备注
   → 订单确认 → 支付 → 锁柜 → 下发取件码
   → 开柜存入起点 → 待转运 → 转运中 → 已送达待取
   → 取件 → 已完成
```

> 说明：行李寄送的"转运中""已送达待取"状态来源依赖小铁寄送配套接口，若一期不具备，寄送上线策略需调整（见待确认事项）。

## 七、用户端页面详细规格

> 每个页面为独立规格，可直接拆分为模块 PRD 供原型开发。所有页面共享第四章全局规则与第五章数据对象。

### 7.1 登录授权

**页面目标**：完成微信授权登录与手机号授权，建立用户身份，作为订单归属依据。

**触发时机**：进入需身份的操作（下单、我的订单）时按需拉起，非强制首屏登录。

**UI 组件与字段**：微信授权按钮、手机号授权按钮、用户协议与隐私政策勾选/链接。

**交互逻辑**：授权成功→建立/绑定 user；拒绝授权→提示影响并可稍后再授权。

**业务规则**：手机号合规采集、脱敏存储；协议隐私需展示并同意。

**关联数据对象**：用户 User。

---

### 7.2 首页

**页面目标**：作为入口，让用户快速选择行李寄存或行李寄送，并就近发现可用柜机。

**区块结构**：
1. 顶部品牌区（产品名 + slogan"放下行李，轻松游湖"）；
2. 双业务入口卡片（行李寄存、行李寄送）；
3. 附近柜机区（基于定位的就近机柜列表）；
4. 进行中订单提醒（有进行中订单时置顶/悬浮，可快捷进入）；

| 小铁柜机控制接口 | **待提供（关键路径）** | 锁柜、开柜、取件码，决定 V1 闭环 |
| 小铁寄送配套接口 | 待确认 | 决定行李寄送是否一期上线 |

**两个必须优先确认的接口问题**：① 取件码由小铁生成回传还是公司生成后登记；② 柜机操作同步返回还是异步回调。

**只读数据接口关键约束（开发须知）**：订单状态仅 processing/completed；无取件码字段；金额单位分；时间范围≤7天；单页≤500；skip≤2000；拉取式同步需定时轮询。
