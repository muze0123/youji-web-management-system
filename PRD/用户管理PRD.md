## 一、模块概述

### 1.1 模块目标

| 场景 | 目标 |
|------|------|
| 客服支撑 | 客诉时按手机号/昵称快速查到用户，查看其订单历史与状态，辅助解决问题 |
| 风控处置 | 发现恶意/异常用户（恶意占柜、纠纷、违规），可封禁使其不能下单 |
| 运营洞察 | 掌握用户规模、新增趋势等基础数据，支撑运营 |

### 1.2 与订单管理的分工（按确认）

| 模块 | 视角 |
|------|------|
| 用户管理 | **以用户为中心**：一个用户的基本信息 + 全部订单 + 风控状态，看这个人的全貌 |
| 订单管理 | **以订单为中心**：单笔订单的字段、状态、处理 |

> 两模块互链：用户详情可看该用户全部订单并跳订单详情；订单详情的用户可反查用户详情。

### 1.3 页面构成

| 页面 | 说明 |
|------|------|
| 用户列表 | 概览条 + 筛选 + 用户表格 + 分页 |
| 用户详情 | 用户信息 + 风控状态 + 该用户订单列表 |
| 封禁处置弹窗 | 封禁/解封操作（限管理员） |

### 1.4 设计与布局

遵循 `design.md`：后台三段布局，侧边导航"用户管理"高亮；主内容区 = 面包屑 + 标题 + 概览条 + 筛选 + 用户表格 + 分页。视觉、组件、语义色引用 design.md；用户ID/手机号/时间用 mono。

## 二、隐私与权限口径（重要）

### 2.1 手机号脱敏（按确认）

| 角色/场景 | 手机号展示 |
|-----------|-----------|
| 默认（列表、详情） | **脱敏展示**（138\*\*\*\*8888） |
| 客服角色 | 可查看**完整手机号**（因需联系用户） |

> 「常见逻辑补充」客服查看完整手机号的动作建议**留痕**（谁在何时查看了某用户完整号码），符合隐私合规与审计要求。是否单独设"客服"角色或在现有角色上加"查看完整号码"权限，见待确认。

### 2.2 操作权限（按确认）

| 操作 | 管理员 | 运营员 |
|------|--------|--------|
| 查看用户列表/详情 | ✅ | ✅ |
| 查看完整手机号 | 按客服角色/权限 | 按客服角色/权限 |
| **封禁/解封用户** | ✅ | ❌ 不可 |

> 封禁类处置**仅管理员**可操作，运营员只读。

## 六、页面三：封禁处置弹窗（仅管理员）

### 6.1 目标

管理员对异常用户封禁或解封，并留痕。

### 6.2 封禁弹窗字段

| 字段 | 必填 | 说明 |
|------|------|------|
| 目标用户 | 自动带入 | 用户ID/昵称 |
| 封禁原因 | 必填 | 如：恶意占柜、纠纷、违规、其他 |
| 封禁时长 | 必填 | 「常见逻辑补充」永久 / 指定时长（如7天/30天） |
| 备注 | 选填 | 详细说明 |

### 6.3 交互

| 操作 | 响应 |
|------|------|
| 提交封禁 | 二次确认 → 用户状态置"已封禁" + 留痕（操作人/时间/原因/时长）→ 该用户小程序端不能下单 |
| 解封 | 二次确认 → 状态置"正常" + 留痕 |
| 取消 | 关闭弹窗 |

> 封禁/解封均需**二次确认 + 留痕**，留痕不可删，供审计与纠纷追溯。

## 七、业务规则

- **手机号脱敏**：默认脱敏展示，客服角色可查看完整手机号并留痕；
- **封禁仅管理员**：运营员只读，不能封禁/解封；
- **封禁效果**：用户被封禁后，小程序端扫码下单被拦截并提示（如"账号异常，请联系客服"）；解封后恢复；
- **数据分工**：用户管理以用户为中心，订单管理以订单为中心，双向互链；
- **运营洞察轻量**：本模块只做用户规模/新增概览，深度分析归数据概览；
- **处置留痕**：封禁/解封、查看完整手机号等敏感操作留痕，供审计；
- **V2 留后**：用户标签/备注、消息触达、注销审核、会员体系等一期不做。
