## 一、模块概述

### 1.1 模块目标

集中管理售后与异常单：接收用户端售后申请、运营标记异常、系统生成异常，流转处理、登记处理结果、留痕，确保售后闭环与三条 P0 异常底线（锁柜失败、取件超时补费、开柜异常）有跟踪、有证据。

### 1.2 单据来源（保留原三种）

| 来源 | 说明 |
|------|------|
| 用户售后 | 用户端提交售后单自动生成（一期 5 类售后，见 2.3）；与用户端售后单为同一实体两端视图 |
| 运营标记 | 运营在订单列表"标记异常"生成 |
| 系统生成 | 系统检测到锁柜失败等异常自动生成 |

> 售后列表 = 三种来源合集，用"来源"字段区分；不拆分为多个模块，运营在一处处理所有售后与异常。

### 1.3 页面构成

| 页面 | 说明 |
|------|------|
| 售后单列表 | 筛选 + 列表 + 分页 |
| 售后单详情/处理面板 | 抽屉/弹窗，查看与处理（含处理结果登记） |
| 标记异常/新建弹窗 | 从订单列表触发，运营生成异常单 |

### 1.4 入口

- 一级菜单"订单管理" → 二级"售后列表"；
- 工作台待办（待处理售后、锁柜失败、售后待处理）带筛选跳入；
- 订单列表"标记异常"生成单；订单详情"查看关联售后/工单"跳入。

### 1.5 设计与布局

遵循 `design.md`：后台三段布局，侧边"订单管理>售后列表"高亮；主内容区 = 面包屑 + 标题 + 筛选区 + 列表 + 分页。单号/订单号/金额/时间用 mono。

## 二、状态、类型与来源

### 2.1 售后单状态（升级：新增"待用户确认"）

| 状态 | 含义 | Badge 色 |
|------|------|----------|
| 待处理 | 新建未受理 | warning |
| 处理中 | 已受理处理中 | info |
| 待用户确认 | 运营给出方案/结果，等待用户确认（用户售后来源专有） | primary |
| 已处理/已完成 | 处理完成 | success |
| 已驳回 | 驳回申请（附原因） | danger |
| 已关闭 | 关闭归档 | ink-muted 灰 |

流转：待处理 → 处理中 →[待用户确认]→ 已完成 / 已驳回 / 已关闭。

> "待用户确认"与用户端售后进度同步（用户在售后单详情确认接受/不接受）；运营标记/系统生成的纯异常单可不经此状态，直接处理→关闭。

### 2.2 异常类型（保留原有）

延误、损坏、遗失、取件码/存储码错误、支付异常、锁柜失败（P0）、开柜异常（P0）、其他。

### 2.3 售后类型（升级：与用户端 5 类打通）

| 售后类型 | 说明 | 对应处理结果（常见） |
|----------|------|----------------------|
| 退款申请（含费用争议） | 用户申请退款 | 同意退款/部分退款/驳回 |
| 物品损坏/丢失 | 行李损坏或丢失 | 赔付/部分退款/驳回 |
| 开柜/取件问题 | 存储码失效/柜门故障（最紧急） | 协助开柜/已解决 |
| 寄送问题 | 延误/送错/未收到 | 部分退款/赔付/已解决/驳回 |
| 其他 | 其他 | 视情况 |

> 售后类型（用户售后来源）与异常类型（运营/系统来源）并存，按来源展示对应分类。

## 四、页面二：售后单详情/处理面板（升级核心）

### 4.1 目标

查看完整信息、处理流转、**登记处理结果（含线下退款/赔付金额）**、留痕。抽屉呈现。

### 4.2 内容分组（升级）

| 分组 | 内容 |
|------|------|
| 单据信息 | 单号、关联订单、类型、来源、状态、创建/更新时间 |
| 关联订单摘要 | 订单号、业务类型、柜机、订单状态、（如有）实付/退款（点击进订单详情） |
| 用户诉求（用户售后来源） | 售后类型、问题描述、图片、期望结果、用户联系方式 |
| 处理记录 | 处理备注留痕列表，每条含操作人+时间+内容，可追加、不可删 |
| **处理结果登记（升级）** | 处理结果选择 + 线下退款/赔付金额与方式记录（见 4.4） |
| 处理操作 | 变更状态、添加备注、指派处理人、提交处理结果、转待用户确认 |

### 4.3 处理操作（升级）

| 操作 | 说明 |
|------|------|
| 受理 | 待处理 → 处理中 |
| 添加处理备注 | 追加留痕（操作人+时间），不可删 |
| 指派处理人 | 分派给运营成员 |
| 登记处理结果 | 选择处理结果并登记金额（见 4.4） |
| 转待用户确认 | 给出方案后转"待用户确认"，同步用户端等待确认（用户售后来源） |
| 完成/驳回/关闭 | 终态流转（驳回需填原因） |

### 4.4 处理结果登记（升级重点 · 一期线下+记录）

| 处理结果 | 登记字段 | 一期落地 |
|----------|----------|----------|
| 同意退款 | 退款金额、退款方式、备注 | **人工线下退款 + 系统记录**（一期资金走小铁账户，不走线上自动退款） |
| 部分退款 | 退款金额、方式、备注 | 同上，线下 + 记录 |
| 赔付 | 赔付金额、方式、备注 | **人工线下赔付 + 系统记录** |
| 协助开柜 | 处理方式（远程/线下）、备注 | 记录处理；远程开柜依赖柜控接口，或线下运维 |
| 驳回 | 驳回原因（必填） | 记录，单据转已驳回 |
| 已解决 | 说明 | 记录，单据转已完成 |

> **重要（一期口径）**：退款/赔付**不走线上自动退款**，运营核实后线下打款/补偿，系统登记金额、方式、时间、操作人，**供对账与审计**（登记金额纳入财务口径，与对账中心关联）。线上自动退款为 V2。

### 4.5 与用户端进度同步

| 后台动作 | 用户端表现 |
|----------|-----------|
| 受理 | 售后单转"处理中" |
| 转待用户确认 | 用户端转"待用户确认"，用户可确认接受/不接受 |
| 用户确认接受 | 单据转"已完成" |
| 用户不接受 | 退回"处理中"，运营再处理 |
| 驳回 | 用户端"已驳回"（展示原因） |

### 4.6 状态

加载中、正常、待用户确认（等待用户）、已关闭（只读，可重开由权限定）。

## 六、业务规则（升级）

- 售后列表含三种来源（用户售后/运营标记/系统生成），一处处理；
- **用户售后来源**与用户端售后单为同一实体两端视图，状态与进度双向同步（含待用户确认）；
- **退款/赔付一期走人工线下 + 系统记录**，登记金额纳入对账与审计，不走线上自动退款（V2 打通）；
- P0 异常（锁柜失败、开柜异常）与开柜取件紧急售后高优先级；
- 处理留痕作证据链，**不可删除，仅可追加**；
- 单据关联订单，双向跳转；
- 权限：管理员与运营员均可处理；「常见逻辑补充」驳回、关闭/重开、退款/赔付金额登记等敏感操作可限管理员或设额度。
