## 一、V2 的整体目标与定位

### 1.1 一期→二期的演进逻辑

一期（V1）是**验证期最小闭环**：依托小铁柜网、公司自研用户端+后台，用真实数据验证游客付费需求，资金走小铁账户、对账人工过渡、用户体系轻量。

二期（V2）是**运营深化期**：在一期验证跑通的基础上，**把业务的主控权和利润收回公司**、**把体验和运营做深**。核心三条主线：

| 主线 | 一句话 |
|------|--------|
| 资金自主 | 资金从小铁账户切到公司账户，自建清结算与自动对账，收回资金主控权 |
| 业务做深 | 行李寄送转运深化（节点调度）、会员与优惠券做用户留存与复购 |
| 运营做强 | 数据大盘、差异化定价、柜机可操作监控、用户精细化运营 |

### 1.2 V2 的前置依赖

V2 多数能力依赖以下前置条件，规划时需同步推进：

- **小铁柜控写接口**（锁柜/开柜/凭证）——一期关键路径，V2 更深依赖；
- **小铁寄送配套接口**（转运状态回传）——决定寄送深化能否做；
- **公司支付通道/商户号**——决定资金切换与自建清结算；
- **一期验证数据**——客单价、单量、寄存/寄送结构，决定 V2 投入方向与定价策略。

## 二、V2 需求全景（按主题）

### 主题 A：资金与结算自主

| 需求 | 一期现状 | V2 目标 |
|------|----------|---------|
| 资金账户切换 | 资金走小铁账户 | 切换到公司账户，收款直接进公司 |
| 自建清结算 | 无 | 公司自建支付商户号、清结算体系 |
| 完整自动对账 | 数据导出人工核对（过渡） | 本地与小铁/支付通道自动对账、差异自动识别、结算 |

### 主题 B：行李寄送深化

| 需求 | 一期现状 | V2 目标 |
|------|----------|---------|
| 转运追踪完整化 | 依赖小铁接口，可能降级为两端状态 | 完整转运轨迹（待转运→转运中→已送达全链路可视）+ 送达通知 |
| 节点调度 | 无（一期柜点存放转运） | 酒店节点、运力管理、（远期）无人车调度，支撑规模化寄送 |

### 主题 C：用户运营与增长

| 需求 | 一期现状 | V2 目标 |
|------|----------|---------|
| 会员体系 | 无（用户体系轻量） | 会员等级、权益、成长体系，提升复购 |
| 优惠券体系 | 一期去掉 | 优惠券发放、核销、营销活动 |
| 用户精细化运营 | 用户管理仅查看+基础风控 | 用户标签/备注、分群、消息触达、注销审核 |

### 主题 D：数据与配置增强

| 需求 | 一期现状 | V2 目标 |
|------|----------|---------|
| 数据大盘 | 基础统计（数据概览） | 复杂 BI、多维分析、经营大盘、可视化看板 |
| 差异化定价 | 全局一套价 | 按柜机/网点/时段差异化定价，精细化收益管理 |

### 主题 E：柜机能力增强

| 需求 | 一期现状 | V2 目标 |
|------|----------|---------|
| 柜机可操作监控 | 只读监控 | 远程开柜、重启、停用维护等操作（依赖小铁写接口），故障处置闭环 |

### 主题 F：平台化（V2 起步 / V3 深化）

| 需求 | 一期现状 | 目标 |
|------|----------|------|
| 多城市多租户 | 单城市验证 | 多城市复制、多租户架构（V2 起步、V3 深化） |

## 三、优先级与建议节奏

综合"对业务的价值 × 依赖成熟度 × 实现成本"，建议 V2 分三批推进：

### 第一批（V2 早期，业务价值高、依赖相对明确）

| 需求 | 理由 |
|------|------|
| 资金账户切换 + 自建清结算 | 收回资金主控权是 V2 核心目标，越早越好；依赖公司商户号 |
| 完整自动对账 | 承接一期过渡方案，资金切换后自然需要 |
| 会员体系 + 优惠券 | 用户留存复购，直接影响增长；业务逻辑明确、不强依赖小铁 |

### 第二批（V2 中期，依赖接口/数据成熟）

| 需求 | 理由 |
|------|------|
| 行李寄送转运追踪完整化 | 依赖小铁寄送接口就绪 |
| 差异化定价 | 依赖一期定价验证数据 |
| 数据大盘 | 依赖一期数据积累 |
| 柜机可操作监控 | 依赖小铁柜控写接口 |

### 第三批（V2 后期 / 转 V3）

| 需求 | 理由 |
|------|------|
| 节点调度（酒店节点/运力/无人车） | 重投入，依赖寄送规模化验证 |
| 用户精细化运营（标签/触达/分群） | 用户量达一定规模后价值才显现 |
| 多城市多租户 | 平台化，转 V3 深化 |

## 四、各主题与一期的衔接要点

| 主题 | 衔接要点 |
|------|----------|
| 资金结算 | 一期对账中心已留"过渡方案"，V2 升级为自动对账；订单/支付数据结构一期已具备 |
| 寄送深化 | 一期寄送 PRD 已按"完整方案+降级标注"设计，V2 补齐转运追踪即可；节点调度对应总纲已留的 V2 模块 |
| 会员/优惠券 | 一期用户 User 预留 member_level 字段；优惠券一期在下单页去掉但保留了扩展位 |
| 数据大盘 | 一期数据概览为基础，V2 在其上做深；口径一致 |
| 差异化定价 | 一期运营配置价格为"全局一套"，V2 扩展生效范围维度（已在运营配置待确认标注） |
| 柜机监控 | 一期只读，V2 加操作能力（已在柜机监控待确认标注 V2 升级） |
| 用户运营 | 一期用户管理仅查看+风控，V2 加标签/触达/注销审核（已在用户管理留 V2） |

## 五、建议优先细化为详版 PRD 的模块

从"业务已明确、不强依赖外部接口、V2 早期就要做"的角度，建议**优先出以下三个模块的详版 PRD**（其余留本规划，落地时再细化）：

| 模块 | 理由 | 建议顺序 |
|------|------|----------|
| **会员体系** | 业务逻辑明确、不依赖小铁接口、直接拉动复购，V2 早期高价值 | 1 |
| **优惠券体系** | 与会员协同做营销，逻辑明确、扩展位已留 | 2 |
| **完整对账与清结算** | 承接一期过渡方案，资金切换的必然需求 | 3 |

> 说明：节点调度、数据大盘、柜机可操作监控等**强依赖接口或一期数据**的模块，建议等依赖成熟、有一期验证数据后再细化，避免现在写太细返工。

## 六、风险与依赖提示

| 风险/依赖 | 说明 |
|-----------|------|
| 小铁接口进展 | 柜控写接口、寄送配套接口的就绪时间直接影响 B、E 主题；需持续跟进 |
| 公司支付资质 | 资金切换与自建清结算依赖公司商户号/支付牌照相关资质 |
| 一期验证结果 | 若一期验证付费需求不达预期，V2 投入方向需重新评估（如是否值得做节点调度重投入） |
| 定价合规 | 差异化定价需注意价格合规与用户感知 |
| 数据合规 | 用户精细化运营（标签/触达）需符合个人信息保护与营销合规要求 |
